# API Application

Elysia API server with PostgreSQL database via Drizzle ORM, Pino logger, dev port 3001.

## Project Structure

- `src/index.ts` тАФ main Elysia app, composes all modules via `.use()`, exports `App` type
- `src/db/` тАФ database connection, types, migration runner
  - `schema/` тАФ table definitions using shared columns from `common.ts` (timestamps, uuidPrimaryKey), re-exported via `index.ts`
- `src/modules/` тАФ one directory per domain concern
- `src/test/` тАФ test context, test DB connection, fixtures directory
  - `fixtures/` тАФ one fixture file per table, composed via `composeFactory()` in `test/fixtures/index.ts`
- `src/utils/` тАФ logger, service-result helpers, shared TypeBox validation schemas, db-error detection
- `drizzle/` тАФ migration files (generated by user only)

## Key Patterns

- **Module factory**: each module exports `create[Name]Route(db)` returning an Elysia instance with a prefix. All modules are composed into the main app via `.use()` in `src/index.ts`.
- **Service layer**: complex business logic is extracted into `service.ts` files within modules. Services accept a `DatabaseConnection` and return `ServiceResult<T>`. Route handlers call services and map results to HTTP responses via `mapError()`.
- **ServiceResult**: discriminated union with `ok(data)` / `fail(code, message)`. Codes: `NOT_FOUND` | `VALIDATION` | `CONFLICT` | `INTERNAL`.
- **Database injection**: all route factories accept a `DatabaseConnection` parameter for testability. Tests provide an auto-rollback transaction; production uses the real connection.
- **Schema conventions**: define tables in `db/schema/[name].ts` using shared columns from `common.ts`. Export types with `InferSelectModel`/`InferInsertModel`. Always re-export from `schema/index.ts`.
- **Schema detection**: pre-commit hook runs `bun schema:check`. Update `SCHEMA_DEPENDENCIES` in `scripts/check-schema-changes.ts` when adding new schema imports.
- **Type exports**: `@meetings-scheduler/api/types` maps to `src/db/schema/index.ts`. Other packages import types via this export.
- **Shared validation schemas**: reusable TypeBox schemas live in `utils/schemas.ts`. Route modules import from there instead of defining inline.

## Module Types

- **Simple**: `index.ts` (route handler) + test file тАФ direct CRUD, no service layer needed.
- **With service**: adds `service.ts` for business logic extraction. Route handler delegates to service, maps `ServiceResult` via `mapError()`.
- **Complex**: adds specialized files тАФ background-task, event-emitter, payload-assembly, external client, etc.

## Testing

- Import `{ test, expect, describe }` from `src/test/test-context.ts` (never from vitest directly).
- Context provides `tx` (auto-rollback transaction), `factory` (bound fixtures), `db`.
- Fixtures: one file per table using `createFixture()` from `@meetings-scheduler/drizzle-fixtures`, composed via `composeFactory()` in `test/fixtures/index.ts`.
- Tests collocated as `[name].test.ts`.

## Database Commands

All commands run from the monorepo root:

| Command                  | Description                                             |
| ------------------------ | ------------------------------------------------------- |
| `bun db:up`              | Start PostgreSQL container                              |
| `bun db:down`            | Stop PostgreSQL container                               |
| `bun db:push:safe`       | **ALWAYS USE** тАФ push schema with auto-snapshot         |
| `bun db:push`            | Push schema (no snapshot тАФ avoid)                       |
| `bun db:generate`        | Generate migration (**USER ONLY** тАФ never run this)     |
| `bun db:generate:custom` | Create empty migration file (**USER ONLY**)             |
| `bun db:migrate`         | Run pending migrations                                  |
| `bun db:migrate:status`  | Check migration status                                  |
| `bun db:seed`            | Seed database with test data                            |
| `bun db:base`            | Create protected baseline snapshot (never auto-deleted) |
| `bun db:snapshot [name]` | Create instant snapshot (auto-rotates old ones)         |
| `bun db:snapshot:list`   | List all snapshots with status                          |
| `bun db:restore [name]`  | Restore from snapshot (latest if no name)               |
| `bun db:reset`           | Return to protected baseline                            |
| `bun db:undo`            | Revert last push instantly                              |
| `bun db:clean`           | Restore baseline + remove ALL snapshots                 |

**Snapshot workflow**:

- Run `bun db:base` before major experiments
- Use `bun db:push:safe` during development (auto-snapshots before each push)
- Use `bun db:undo` or `bun db:reset` on failure
- NEVER run `bun db:generate` тАФ only the user creates migrations
