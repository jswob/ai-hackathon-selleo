## 1. Overview

The RuleBuilder manages constraint rules with IF-THEN logic, serving as a semantic bridge between structured data and human-readable UI. All text is generated by a single source of truth: `rule-builder.text-descriptor.ts`.

**Key Capabilities**: Cross-role comparisons, position-aware constraints, type-safe transformations, adaptive input controls, cascading field updates, schema-based validation.

## 2. Component Hierarchy

```
RuleBuilder (orchestrator)
├── ConditionBlock[] (IF clauses - purple)
│   ├── SelectInput (source), TraitSelector, OperatorSelector, ComparisonInput
└── RequirementBlock (THEN clause - blue)
    ├── TraitSelector, SelectInput (targetPositions), SelectInput (enforcement)
    ├── OperatorSelector, ComparisonInput
```

**RuleBuilder**: Manages rule state, computes text descriptors (`createRuleTextDescriptor()`), handles cascading updates, distributes errors.
**ConditionBlock**: IF clauses, manages cascading resets, conditionally shows position operators.
**RequirementBlock**: THEN clause, always applies to candidate, displays target position and enforcement controls.
**OperatorSelector**: Groups operators by type, filters by trait compatibility, uses prefixed values (`value:eq`, `position:gt`).
**ComparisonInput**: Polymorphic input adapting to operator type and trait data type, implements pattern validation.

Source: `RuleBuilder.tsx`, `ConditionBlock.tsx`, `RequirementBlock.tsx`, `OperatorSelector.tsx`, `ComparisonInput.tsx`

## 3. Data Model

**SourceType**: `'candidate'` (being assigned) | `'position-1'` (first role, API position 0) | `'position-2'` (second role, API position 1)

**ComparisonValue** (discriminated union):

```typescript
{ type: 'value', value: string | string[] | null }
| { type: 'position', position: 0 | 1, traitDefinitionId: string }
```

**ConditionState**: IF clause with `source`, `traitDefinitionId`, `operator` (prefixed), `comparison`.
**RequirementState**: THEN clause with `traitDefinitionId`, `operator` (prefixed), `comparison`.
**RuleBuilderState**: Complete rule with `targetPositions` ('all'|'[0]'|'[1]'), `enforcement` (HARD|SOFT), `weight`, `conditions[]`, `requirement`.

**Trait Data Types** (influence operators and input rendering):

- **string**: eq, neq, in, not_in, exists, not_exists
- **number**: adds gt, gte, lt, lte
- **boolean**: adds is_true, is_false
- **enum**: same as string but renders as select

## 4. Semantic Bridge

The text descriptor system (`rule-builder.text-descriptor.ts`) is the single source of truth for all human-readable content, transforming structured data into natural language for visual presentation and accessibility.

**Benefits**: Consistency, accessibility (complete aria-label sentences), maintainability (one file for semantic changes), testability.

**Grammatical Adaptation**:

- Conditions use present tense ("is equal to"), requirements use infinitive ("be equal to")
- Possessive pronouns adapt to source ("their" for candidate, "its" for positions)
- First condition gets "if", subsequent get "and if"
- "then" omitted when no conditions exist
- Existence operators omit connector words ("is"/"be")

## 5. Operator System

**Operator Prefixing**: `value:eq` (static value) vs `position:gt` (cross-role comparison). Same base operator, different semantics.

| Trait Type  | Value Operators                                           | Position Operators        |
| ----------- | --------------------------------------------------------- | ------------------------- |
| **string**  | eq, neq, in, not_in, exists, not_exists                   | eq, neq                   |
| **number**  | eq, neq, gt, gte, lt, lte, in, not_in, exists, not_exists | eq, neq, gt, gte, lt, lte |
| **boolean** | eq, neq, is_true, is_false, exists, not_exists            | ❌ None                   |
| **enum**    | eq, neq, in, not_in, exists, not_exists                   | eq, neq                   |

**Note**: Existence (exists, not_exists, is_true, is_false) and multi-value (in, not_in) operators do not support position comparisons.

**Availability**:

- **Conditions**: Position operators when source is `position-1` or `position-2` (not `candidate`)
- **Requirements**: Position operators when targetPositions is `[0]` or `[1]` (not `all` — ambiguous)
- **Enum comparisons**: Uses same trait on other role; non-enum lets user select trait

## 6. Target Positions

| Mode    | Applies To       | Position Operators | Comparison Position    |
| ------- | ---------------- | ------------------ | ---------------------- |
| **all** | All participants | ❌ Not available   | N/A                    |
| **[0]** | First role only  | ✅ Available       | Compares to position 1 |
| **[1]** | Second role only | ✅ Available       | Compares to position 0 |

**Cascading Effects**:

- **TO 'all' with position operator**: Reset to value operator, clear comparison (position comparison invalid)
- **FROM 'all' to position-specific**: Enable position operators
- **BETWEEN [0] ↔ [1]**: Flip comparison position (0 ↔ 1) to reference "other" role

## 7. Enforcement Modes

**HARD (must)**: `enforcement: 'HARD', weight: 0` — Absolute requirements, violations invalid, solver cannot override.
**SOFT (should)**: `enforcement: 'SOFT', weight: 5` — Optimization objectives, violations permitted, contributes penalty.

Weights auto-derived via `getWeightForEnforcement()`.

## 8. Polymorphic Input & Cascading

**ComparisonInput Adaptation** (three-tier decision tree):

1. Operator group (value/position) → static value vs position reference input
2. Operator requirements → exists/not_exists render nothing
3. Input type (single/multi + trait data type) → text/number/select/multi-select

**Disabled when** no trait or operator selected. **Pattern validation** blocks invalid characters for in/not_in: `COMMA_SEPARATED_NUMBERS_PATTERN` / `COMMA_SEPARATED_STRINGS_PATTERN`.

**Cascading Triggers**:
| Trigger | Reset Fields | Rationale |
| ------------------------------------------- | ------------------------------------- | ----------------------------------------- |
| Condition source change | operator, comparison | Sources affect operator availability |
| Condition/requirement trait change | operator, comparison | Traits have incompatible operators |
| Condition/requirement operator change | comparison (restructured) | Operator group determines structure |
| targetPositions → 'all' w/ position op | operator (to value), comparison | Position comparison invalid |
| targetPositions w/ position comparison | comparison.position (flip 0↔1) | References "other" role |
| Enforcement change | weight (derived) | Auto-update (HARD→0, SOFT→5) |

Source: `ConditionBlock.tsx`, `RequirementBlock.tsx`, `RuleBuilder.tsx`

## 10. Validation & Error Handling

The validation system ensures data integrity through three layers: schema-based validation (Zod), context-aware validation (role count), and error extraction/mapping.

### Semantic Validation

Validates comparison values match operator type using `.superRefine()` in `conditionStateSchema` and `requirementStateSchema`:

- **Early return** on empty operator prevents premature errors
- **parseOperator()** removes prefix (`value:eq` → `eq`)
- **Type-discriminated** validation checks comparison type before value structure
- **Custom messages**: `z.string().uuid({ message: 'Trait is required' })` instead of "Invalid UUID"

**Examples**: Single-value operators (`eq`) require non-empty value; multi-value operators (`in`) require array with at least one value.

### Position Existence Validation

Schema factory `createRuleBuilderSchema(roleCount)` adapts validation to current task type configuration. Validates:

- **Target positions**: `[0]` and `[1]` reference defined roles
- **Condition sources**: `position-1` and `position-2` exist
- **Position comparisons**: `comparison.position` values valid in conditions and requirements

Context-aware: 1-role task rejects `[1]` targets; 2-role task accepts them.

### Error Extraction & Mapping

Transforms Zod errors into component-specific props via `extractRuleErrors(error)`:

```typescript
type ConditionErrors = {
	source?: string;
	traitDefinitionId?: string;
	operator?: string;
	comparisonValue?: string;
	comparisonPosition?: string;
	comparisonTrait?: string;
};
type RequirementErrors = {
	/* same fields except source */
};
interface RuleErrors {
	targetPositions?: string;
	conditions: Record<number, ConditionErrors>;
	requirement?: RequirementErrors;
}
```

**First error wins**: Multiple Zod issues for same field show only first message.
**Batch helper**: `validateAllRules(rules, roleCount)` returns indexed error map or null.

### Error Flow Through Components

**Form Level** (`useAddTaskTypeForm.ts`):

```typescript
const [ruleErrors, setRuleErrors] = useState<Record<number, RuleErrors>>({});
function clearRuleErrors(ruleIndex: number) {
	/* delete from map */
}
function handleSubmit(e) {
	setRuleErrors({}); // Clear before validation
	const errors = validateAllRules(rules, roleCount);
	if (errors) {
		setRuleErrors(errors);
		return;
	}
	// ... proceed
}
```

**Form Fields** (`TaskTypeFormFields.tsx`):

```typescript
function updateRule(index, updates) {
	clearRuleErrors(index); // Auto-clear on edit
	setRules(/* ... */);
}
```

**Component Props**: RuleBuilder receives `errors?: RuleErrors`, distributes to children:

- `<SelectInput error={errors?.targetPositions} />`
- `<ConditionBlock errors={errors?.conditions[index]} />`
- `<RequirementBlock errors={errors?.requirement} targetPositionsError={errors?.targetPositions} />`

Children wire errors to input fields: `<ComparisonInput error={errors?.comparisonValue || errors?.comparisonPosition || errors?.comparisonTrait} />`

### Error Clearing Strategy

**Phase 1 (Clear on Submit)**: All errors cleared at submission start, fresh validation results shown.
**Phase 2 (Auto-Clear on Edit)**: When ANY field in a rule changes, all errors for that rule clear immediately.

**Rationale**: No premature errors, no stale errors, no flashing, consistent UX.

## 11. Key Examples

### Example 1: Simple Value Comparison

**Scenario**: All participants must have availability equal to "high".

**Structure**:

```
targetPositions: 'all', enforcement: 'HARD', conditions: []
requirement: { traitDefinitionId: (availability UUID), operator: 'value:eq', comparison: { type: 'value', value: 'high' } }
```

**Semantics**: "Availability of participants must be equal to high"
**Behavior**: No conditions = rule always applies. HARD enforcement = mandatory for all candidates.

### Example 2: Position Comparison Rule

**Scenario**: First role's experience level must be greater than or equal to second role's experience level.

**Structure**:

```
targetPositions: '[0]', enforcement: 'HARD', conditions: []
requirement: {
  traitDefinitionId: (experience_level UUID), operator: 'position:gte',
  comparison: { type: 'position', position: 1, traitDefinitionId: (experience_level UUID) }
}
```

**Semantics**: "Experience_level of first role must be greater than or equal to experience_level of second role"
**Behavior**: Applies only when assigning to first role. Compares candidate's trait to existing second role participant. Enables cross-role constraints.

**Enforcement Comparison**:

- **HARD (must)**: Absolute requirement, violations invalid, no solution if unsatisfiable. Use for: legal/safety/contractual obligations.
- **SOFT (should)**: Optimization objective, violations permitted, adds penalty (weight × count). Use for: preferences, desired pairings, optimization goals.
- **Combined**: Solver satisfies all HARD, then minimizes SOFT penalties. Higher weight = more important.
